# LAN BeamDrop - 需求与架构总览 (Product & Architecture Specs)

本文档是 LAN BeamDrop 项目的**核心知识总线**，融合了产品需求 (PRD)、系统设计架构以及最高优先级的工程开发准则。

---

## 一、 产品需求与核心定位 (PRD)
LAN BeamDrop 是一款专注于局域网内多设备协同的“零安装”传输工具。

### 1. 核心价值
*   **零安装门槛**：PC 端启动服务端后，局域网内任何设备通过浏览器访问即用。
*   **全能传输聚合**：将文本、图片、大文件统一收拢至类似聊天流的时间线（Timeline）中。
*   **多端身份识别**：打破传统局域网工具“分不清谁是谁”的困境，自动识别区分不同设备的输入。

### 2. 功能形态
*   **统一聊天流**：左右侧隔离（当前设备在右，其他设备在左），文本和文件严格按时间戳混排。
*   **极速大文件传输**：支持批量、超大文件流式传输。具备上传防抖、上传取消机制，并实时反馈进度和上传网速。
*   **多端剪切板流转**：支持网页内文本的一键复制与手动发送共享，保留历史文本记录并随用随取，杜绝后台静默监听以实现完美隐私保护与低耗能。

## 二、 系统架构与模块设计

本系统采用极简的高性能 **B/S 架构**，坚决不引入任何数据库：
*   **后端 (Node.js + Express + Multer + ws)**：专注于 RESTful API、底层流式文件高速 IO 以及 WebSocket 全双工实时通信。
*   **前端 (Vanilla ES6 Modules + HTML5 + CSS3)**：零编译依赖，利用模块化设计、WebSocket 增量推送与 Diff DOM 理念，实现平滑的消息瀑布流。

### 1. 数据存储与生命周期
*   **文本历史**：驻留于服务端内存，按容量截断（最多 50 条），重启即焚。
*   **文件系统**：所有传输文件落盘至 `config.shareDir`。文件列表通过即时读取本地目录动态生成（并严格过滤如 `.chunks` 等隐藏系统缓存目录）。
*   **残缺文件隔离**：所有上传中的大文件切片缓存于隔离目录，并在合并前强制追加 `.uploading` 后缀，防止其它设备误读，具备自动销毁机制。

### 2. 核心性能优化
*   **大文件分片并发 (Chunked Upload)**：彻底解决 Node.js 内存溢出问题。前端文件分片后并发上传，并内建自动静默重试机制（防御网络抖动），服务端收集后合并。
*   **WebSocket 增量推送**：全面废弃了传统的 Fetch HTTP 短轮询，全面拥抱 WebSocket 增量实时推送，极大降低了无数据时的服务端空闲性能开销。
*   **热重载阻断机制**：通过在接收流前校验 `Content-Length`，实现对超限文件的瞬间无损阻断。

## 三、 工程铁律与跨平台准则 (Ironclad Rules)

所有参与此项目的开发者与 AI Agent，**必须无条件遵守**以下准则：

### 1. 绝对的跨平台兼容 (Win/Mac/Linux)
本系统必须能在三大系统上实现 Node.js 源码的无缝部署与运行。
*   **文件路径**：严禁硬编码绝对路径（如 `C:\`、`/tmp`）。必须使用 `path.join()`、`os.tmpdir()`、`os.homedir()`。
*   **文件名安全防线**：
    *   **防目录穿越**：所有外部传入的文件名必须经过 `path.basename()` 净化。
    *   **跨系统死机防御**：Mac/Linux 客户端可能上传带特殊字符（如 `?`, `*`, `:`）的文件名，落盘 Windows 时会引发 `EINVAL` 崩溃。必须使用 `/[<>:"\/\\|?*]/g` 正则将文件名彻底跨平台净化。
*   **系统调用**：避免特定系统命令（如 Windows 的 `wmic`），如果必须使用，必须结合 `os.platform()` 给出安全 Fallback。

### 2. 行为级集成测试 (Behavioral Integration Testing)
*   仅仅测试“配置数据被成功写入 JSON”是**不合格的**。
*   如果修改了影响系统行为的配置项（如 `maxFileSize`），你必须编写真实的集成测试：例如构造一个超过上限的 Payload 攻击服务端，并断言服务端成功抛出了 `413 Payload Too Large` 错误。

### 3. 正向设计大于防御性补丁
*   面对 Bug，必须找到引发异常的最底层原因（Root Cause）。严禁为了掩饰报错而在表层写 `if (err) ignore` 的创可贴代码。
*   前端 UI 的状态更新应以服务端 API 的最终响应为准，杜绝虚假的状态反馈。

### 4. 代码提交规范 (Git Commit Format)
所有代码提交必须极其精简（不能啰嗦），并严格遵循以下模板：
```text
[NEW/ENH/FIX]: <精简的一句话描述>

WHY:
<解释为什么做这个改动>

WHAT:
<解释做了哪些具体的代码层改动>

TEST:
<解释如何验证了这个改动>
```

### 5. 智能体排查与提交铁律 (Agent Debugging & Commit Iron Rule)
**禁止“灯下黑”与盲目钻牛角尖。**
在处理前后端交互 Bug（如状态不一致、UI 越权显示）时，**严禁直接修改底层网络或鉴权代码**。
1. **必须先寻找“分水岭”**：通过查阅 API 日志或抓包，明确服务端返回给前端的 Payload 到底是什么。
2. **端到端闭环验证**：确认是后端的返回有误，还是前端的 JS 解析、甚至 CSS 渲染丢失（注意 CSS 的静默失效特性）引起的错误。
3. **不要预设立场**：即使后端确实存在潜在的隐患逻辑，只要“分水岭”证明当前 Bug 是前端渲染引起的，就优先修复前端。在确认界限之前，不要过度发散修改。
4. **提交前强制全局 Self-Review（不要等用户指出）**：在执行 `git add .` 之前，必须主动使用 `git diff` 和 `git status` 审查当前工作区的每一行改动。**严禁**将用于排查问题的测试脚本（如 `test-ip.js`）、临时日志打印（如 `console.log`）以及多余的中间文件提交入库。时刻保持代码库的极致整洁。

### 6. 智能体交互与工作流纪律 (Agent Workflow Protocol)
**禁止自作主张、抢跑施工。**
所有 Agent 在执行任何涉及代码修改的任务时，必须严格遵守以下串行工作流：
1. **需求对齐与答疑**：必须先与开发者深入讨论，正面且完整地回答开发者的所有疑问。
2. **输出方案与计划**：在消除疑问后，输出结构化的技术方案和实施计划（Implementation Plan）。
3. **请示动工**：明确向开发者询问“是否可以动工”。
4. **获批执行**：只有在开发者明确同意后，才能调用工具开始修改文件或执行命令。绝不允许在回答问题的同一轮对话中夹带偷偷修改文件的操作。
5. **强制测试与提交通知**：代码修改完毕后，**严禁直接 Git Commit！**必须先进行完整的测试（包括提示开发者进行端到端手工测试）。测试确认无误后，必须向开发者询问“是否可以提交（Commit）”。得到明确许可后方可入库。
