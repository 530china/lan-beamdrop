# LAN BeamDrop - 需求与架构总览 (Product & Architecture Specs)

本文档是 LAN BeamDrop 项目的**核心知识总线**，融合了产品需求 (PRD)、系统设计架构以及最高优先级的工程开发准则。

---

## 一、 产品需求与核心定位 (PRD)
LAN BeamDrop 是一款专注于局域网内多设备协同的“零安装”传输工具。

### 1. 核心价值
*   **零安装门槛**：PC 端启动服务端后，局域网内任何设备通过浏览器访问即用。
*   **全能传输聚合**：将文本、图片、大文件统一收拢至类似聊天流的时间线（Timeline）中。
*   **多端身份识别**：打破传统局域网工具“分不清谁是谁”的困境，自动识别区分不同设备的输入。

### 2. 功能形态
*   **统一聊天流**：左右侧隔离（当前设备在右，其他设备在左），文本和文件严格按时间戳混排。
*   **极速大文件传输**：支持批量、超大文件流式传输。具备上传防抖、上传取消机制，并实时反馈进度和上传网速。
*   **剪切板双向共享**：支持移动设备和 PC 物理剪切板之间的文本高速流转。

---

## 二、 系统架构与模块设计

本系统采用极简的高性能 **B/S 架构**，坚决不引入任何数据库：
*   **后端 (Node.js + Express + Multer)**：专注于 RESTful API 和底层流式文件高速 IO。
*   **前端 (Vanilla JS + HTML5 + CSS3)**：零编译依赖，利用 Fetch 轮询与 Diff DOM 理念，实现平滑的消息瀑布流。

### 1. 数据存储与生命周期
*   **文本历史**：驻留于服务端内存，按容量截断（最多 50 条），重启即焚。
*   **文件系统**：所有传输文件落盘至 `config.shareDir`。文件列表通过即时读取本地目录动态生成。
*   **残缺文件隔离**：所有上传中的文件会被强制追加 `.uploading` 后缀进行隔离，防止其它设备误读，并具备 24 小时自动销毁机制。

### 2. 核心性能优化
*   **流式传输 (Stream)**：文件 IO 采用管道 (pipe) 或 multer 原生流，禁止全量载入内存。
*   **热重载阻断机制**：通过在接收流前校验 `Content-Length`，实现对超大文件的瞬间无损阻断。

## 三、 工程铁律与跨平台准则 (Ironclad Rules)

所有参与此项目的开发者与 AI Agent，**必须无条件遵守**以下准则：

### 1. 绝对的跨平台兼容 (Win/Mac/Linux)
本系统必须能在三大系统上实现 Node.js 源码的无缝部署与运行。
*   **文件路径**：严禁硬编码绝对路径（如 `C:\`、`/tmp`）。必须使用 `path.join()`、`os.tmpdir()`、`os.homedir()`。
*   **文件名安全防线**：
    *   **防目录穿越**：所有外部传入的文件名必须经过 `path.basename()` 净化。
    *   **跨系统死机防御**：Mac/Linux 客户端可能上传带特殊字符（如 `?`, `*`, `:`）的文件名，落盘 Windows 时会引发 `EINVAL` 崩溃。必须使用 `/[<>:"\/\\|?*]/g` 正则将文件名彻底跨平台净化。
*   **系统调用**：避免特定系统命令（如 Windows 的 `wmic`），如果必须使用，必须结合 `os.platform()` 给出安全 Fallback。

### 2. 行为级集成测试 (Behavioral Integration Testing)
*   仅仅测试“配置数据被成功写入 JSON”是**不合格的**。
*   如果修改了影响系统行为的配置项（如 `maxFileSize`），你必须编写真实的集成测试：例如构造一个超过上限的 Payload 攻击服务端，并断言服务端成功抛出了 `413 Payload Too Large` 错误。

### 3. 正向设计大于防御性补丁
*   面对 Bug，必须找到引发异常的最底层原因（Root Cause）。严禁为了掩饰报错而在表层写 `if (err) ignore` 的创可贴代码。
*   前端 UI 的状态更新应以服务端 API 的最终响应为准，杜绝虚假的状态反馈。

### 4. 代码提交规范 (Git Commit Format)
所有代码提交必须极其精简（不能啰嗦），并严格遵循以下模板：
```text
[NEW/ENH/FIX]: <精简的一句话描述>

WHY:
<解释为什么做这个改动>

WHAT:
<解释做了哪些具体的代码层改动>

TEST:
<解释如何验证了这个改动>
```
